# 绿色物流碳足迹优化平台

基于 Streamlit 构建的多页面物流优化系统，面向多场馆、多物资类型、多车型车队的配送规划场景，提供从仓库配置、场馆导入、物资需求解析、车辆编组、路径优化到碳排放分析与结果导出的完整业务闭环。

项目当前聚焦赛事物流、会展物流、集中型大型活动保障等典型场景，核心目标是在满足配送需求与车辆容量约束的前提下，降低总运输距离、提升装载利用率，并量化优化前后的碳排放差异。

## 1. 核心能力

1. 多页面业务流
   以 Streamlit 多页面应用串联“仓库设置 -> 场馆录入 -> 物资需求 -> 车辆配置 -> 路径优化 -> 碳排概览 -> 碳排分析 -> 优化结果”全流程。
2. 智能数据导入
   支持 CSV、XLSX、XLS、TXT、JSON 等格式，内置列名别名映射、编码兼容、空值清洗、行数与文件大小安全校验。
3. 场馆地理定位
   支持手工录入经纬度，也支持通过高德地图 API 对地址进行地理编码，并在地图上可视化场馆分布。
4. 物资需求解析
   支持复杂 Excel 表头识别、前置说明行跳过、单位自动换算为 kg、合计列过滤、多类别物资汇总。
5. 异构车队建模
   支持柴油、LNG、HEV、PHEV、BEV、FCEV 等多种车型，并支持设置每类车辆的装载吨位与最大可用数量。
6. 中转仓选址
   基于自定义加权 K-Medoids 逻辑，从全国候选仓库数据集中筛选中转仓位置，降低末端配送距离。
7. 路径优化求解
   基于 OR-Tools 构建 FSMVRP（异构车队车辆路径问题）求解流程，支持容量约束、车型差异、冷启动固定成本与碳排放成本最小化。
8. 分段碳排核算
   按“逐段运输、逐站卸货、载重递减”的方式核算路线碳排放，并纳入季节因子、氢气来源因子和冷启动排放。
9. 可视化分析
   提供地图、指标卡、对比柱图、结构分析图、车辆调度明细表等结果展示方式。
10. 结果导出
    支持导出关键表格结果，便于汇报、复盘和后续数据沉淀。

## 2. 适用场景

本项目适用于以下类型的物流优化任务：

- 大型赛事或活动保障物流
- 多场馆联动配送
- 具备中转仓选址需求的区域配送
- 新能源车队与传统车队碳排对比分析
- 以碳排放最小化为目标的运输组织方案评估

## 3. 业务流程

| 步骤 | 页面 | 说明 |
| --- | --- | --- |
| 1 | 仓库设置 | 配置总仓名称、地址、坐标、容量等基础信息 |
| 2 | 场馆录入 | 批量导入或手工维护场馆清单，并进行地图定位 |
| 3 | 物资需求 | 导入多类别物资需求，自动识别重量并汇总 |
| 4 | 车辆配置 | 选择车型、设置单车载重、最大车辆数 |
| 5 | 路径优化 | 执行距离矩阵构建、中转仓选址、FSMVRP 求解 |
| 6 | 碳排概览 | 从总量、效率、结构等维度查看碳排指标 |
| 7 | 碳排分析 | 对比车型方案、分析敏感因素与减排空间 |
| 8 | 优化结果 | 查看路线地图、车辆调度明细、碳排对比与导出结果 |

## 4. 技术架构

| 层级 | 组件 | 作用 |
| --- | --- | --- |
| 表现层 | Streamlit | 多页面 Web 交互、表单、指标卡、导航 |
| 地图层 | Folium + streamlit-folium | 仓库、场馆、中转仓、路线地图展示 |
| 图表层 | Plotly | 碳排放与优化结果图表分析 |
| 数据处理层 | Pandas + NumPy | 文件读取、清洗、聚合、矩阵计算 |
| 求解层 | OR-Tools | 异构车队 VRP / FSMVRP 优化求解 |
| 选址层 | 自定义 Weighted K-Medoids | 中转仓候选点筛选与聚类分配 |
| 地理服务层 | 高德地图 API | 地理编码、逆地理编码、可选路网距离获取 |
| 缓存层 | SQLite | 地理编码缓存、距离矩阵缓存 |

## 5. 核心算法说明

### 5.1 中转仓选址

项目在 [`utils/clustering.py`](./utils/clustering.py) 中实现了自定义加权 K-Medoids 选址逻辑：

- 输入为场馆经纬度、需求权重、候选仓库坐标集
- 使用 Haversine 球面距离衡量候选点与需求点之间的空间距离
- 以“距离 × 需求权重”的总成本最小化为目标选择中转仓
- 输出最优仓库数、候选仓位置、场馆归属关系及簇权重

与直接使用几何中心不同，该实现要求最终中转仓落在候选仓库数据集中，更贴近真实落地选址。

### 5.2 路径优化

项目在 [`utils/fsmvrp_ortools.py`](./utils/fsmvrp_ortools.py) 中构建了 OR-Tools 求解流程，主要特点如下：

- 车辆池由“车型 + 单车载重 + 最大车辆数”展开为可用车辆集合
- 通过容量维度约束每条路线的总装载量
- 为不同车辆设置不同的单位运输碳成本
- 将冷启动排放设置为固定车辆成本，抑制不必要的派车
- 根据求解时限自动切换局部搜索策略，并启用并行搜索 worker

当前优化目标更接近“总碳排放最小化”而不是简单的“总里程最短”。

### 5.3 分段碳排核算

项目在 [`utils/carbon_calc.py`](./utils/carbon_calc.py) 和 [`utils/vehicle_lib.py`](./utils/vehicle_lib.py) 中定义了车辆参数与碳排计算逻辑：

- 有载运输段采用 `排放强度 × 载重(吨) × 距离(km)` 计算
- 空驶段采用车型对应的空载排放系数
- 新能源车型可叠加季节修正系数
- FCEV 车型支持按灰氢、工业副产氢、绿氢切换排放强度
- 路线按逐站卸货方式递减载重，得到更细粒度的分段排放结果

## 6. 车辆模型

系统内置以下车辆类型：

| 车型标识 | 类型说明 |
| --- | --- |
| `diesel` | 柴油重卡 |
| `lng` | LNG 天然气重卡 |
| `hev` | 柴电混动 HEV |
| `phev` | 插电混动 PHEV |
| `bev` | 纯电动 BEV |
| `fcev` | 氢燃料电池 FCEV |

系统支持以下环境因子：

- 季节：`spring`、`summer`、`autumn`、`winter`
- 氢气来源：`grey`、`byproduct`、`green`

## 7. 数据输入要求

### 7.1 场馆数据

系统默认要求至少具备以下列：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 场馆名称 | 是 | 用于场馆唯一识别与结果展示 |
| 地址 | 是 | 用于地理编码；若已提供经纬度仍建议保留 |
| 经度 | 否 | 可直接跳过地理编码 |
| 纬度 | 否 | 可直接跳过地理编码 |
| 承办项目 | 否 | 用于业务展示 |
| 需求量 | 否 | 可作为简单需求输入 |

内置列名别名映射，例如 `venue`、`venue_name`、`name` 可映射为“场馆名称”，`lng`/`lon` 映射为“经度”，`lat` 映射为“纬度”。

### 7.2 物资需求数据

物资需求页面支持更复杂的 Excel 模板，具备以下处理能力：

- 自动扫描前若干行并识别真实表头
- 自动识别多类别物资列
- 自动过滤“合计 / 总计 / Total”等汇总列，避免重复累计
- 支持 `500g`、`1.2t`、`20kg` 等单位字符串解析并统一换算为 kg

### 7.3 文件约束

项目当前代码内置如下限制：

- 单文件大小上限：10 MB
- 导入数据默认行数上限：200 行
- 支持格式：`.csv`、`.xlsx`、`.xls`、`.txt`、`.json`
- 为兼容 `.xls` 文件，运行环境需安装 `xlrd`

## 9. 运行环境

建议环境如下：

- Python 3.10 或 3.11
- Windows / macOS / Linux
- 推荐使用虚拟环境隔离依赖

## 10. 本地部署

### 10.1 安装依赖

```bash
cd green-logistics-platform
python -m venv .venv
.venv\Scripts\activate
pip install -U pip
pip install -r requirements.txt
```

如果你使用 macOS 或 Linux，请将激活命令替换为：

```bash
source .venv/bin/activate
```

### 10.2 启动应用

```bash
streamlit run app.py
```

启动后默认会在本机浏览器打开 Streamlit 页面。

## 11. 高德地图 API 配置

高德地图 API 不是强制项，但在以下场景中建议配置：

- 地址转经纬度的地理编码
- 经纬度转地址的逆地理编码
- 使用高德路网距离代替本地近似距离

### 11.1 本地开发配置

可在 `.streamlit/secrets.toml` 中添加：

```toml
AMAP_API_KEY = "your_amap_api_key"
```

### 11.2 Streamlit Cloud 配置

在应用 `Settings -> Secrets` 中添加同名配置：

```toml
AMAP_API_KEY = "your_amap_api_key"
```

### 11.3 未配置 API Key 时的行为

未配置高德密钥时，项目仍可运行，但会退化为：

- 地理编码能力受限
- 路径距离主要使用 Haversine 球面距离乘道路折算系数估算
- 逆地理编码仅返回占位文本

## 12. 缓存与数据文件

项目使用 SQLite 进行缓存以减少重复计算和外部 API 调用：

| 文件 | 作用 |
| --- | --- |
| `data/cache/amap_cache.db` | 缓存地理编码、逆地理编码、路径规划结果 |
| `data/cache/distance_cache.db` | 缓存距离矩阵中的点对距离与时长 |
| `data/cache/geocode.json` | 历史缓存文件，可按项目需要保留 |

全国候选仓库数据默认从 `data/national_warehouse_candidates.csv` 或同名 `.xlsx` 文件加载。当前仓库中已提供 Excel 数据文件，可直接使用。

## 13. 输出结果

路径优化完成后，系统会生成以下核心成果：

- 中转仓分配结果
- 每辆车的访问场馆序列
- 路线总里程、干线里程、末端配送里程
- 单车与整体碳排放结果
- 优化前后碳排差异与减排比例
- 地图化路径展示与车辆调度明细表

## 14. 已知约束与说明

为保证使用预期清晰，建议注意以下几点：

- 候选中转仓选址依赖本地候选仓库数据集，不是连续空间自由选址
- 不配置高德 API 时，距离为近似估算值，更适合方案比较与早期评估
- 大规模节点问题在 Streamlit 单进程环境下可能出现求解时间上升
- 当前默认导入行数限制为 200，适合中小规模业务验证与方案演示
- 项目代码对中国经纬度范围做了校验，更适合中国区域场景
